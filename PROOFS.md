# Verification status

This file lists the code in this repository that carries a machine-checked
proof. Code that is not listed here is not verified.

## Scope

- Only the algorithms in the two tables below are verified. Any other file,
  crate, module, or API, including top-level wrappers and algorithms not listed
  (for example AES and HMAC-DRBG), is not verified.
- The proofs do not establish side-channel resistance. The source is written to
  be secret-independent, but this property is not part of the proofs.
- For the HACL\*-derived crates, the Rust wrappers around the verified code are
  not necessarily verified (see §1).

Two verification paths are used:

1. Algorithms verified in F\* as part of HACL\* and compiled to Rust (§1).
2. Rust verified directly with hax, which extracts it to F\* (§2).

## 1. Verified in HACL\*, compiled to Rust

These crates are Rust compiled from HACL\*'s F\*-verified C, shared through the
`libcrux-hacl-rs` crate. The upstream source is HACL\*
(<https://github.com/hacl-star/hacl-star>); the tools are F\* and the formalized
C-to-Rust compiler of Fromherz and Protzenko
(<https://arxiv.org/abs/2412.15042>). The HACL\* proofs establish memory safety,
functional correctness against a high-level specification, and secret
independence (<https://eprint.iacr.org/2017/536>,
<https://arxiv.org/abs/1703.00053>).

| Algorithm | Verified code |
|---|---|
| BLAKE2 | `crates/algorithms/blake2/src/hacl/` |
| ChaCha20-Poly1305 | `crates/algorithms/chacha20poly1305/src/hacl/` |
| Curve25519 (X25519) | `crates/algorithms/curve25519/src/impl_hacl.rs` |
| ECDSA (P-256) | `crates/algorithms/ecdsa/` |
| Ed25519 | `crates/algorithms/ed25519/src/hacl/` |
| HKDF | `crates/algorithms/hkdf/src/hacl.rs` |
| HMAC | `crates/algorithms/hmac/src/hacl/` |
| P-256 (ECDH) | `crates/algorithms/p256/` |
| Poly1305 | `crates/algorithms/poly1305/src/hacl/` |
| RSA-PSS | `crates/algorithms/rsa/src/hacl/` |
| SHA-2 | `crates/algorithms/sha2/src/hacl.rs` |

Only the HACL\*-derived code is verified. Wrappers around it may not be; for
example, the XChaCha20-Poly1305 wrapper in the `chacha20poly1305` crate is not
verified.

## 2. Verified in Rust with hax

These crates are verified by extracting their Rust to F\* with hax
(<https://github.com/cryspen/hax>, <https://eprint.iacr.org/2025/142>). Each
crate's Rust is in `src/` and its F\* proofs are under `proofs/fstar/`. Where a
status document exists it records the per-function result and is generated by
the crate's `proofs/generate_verification_status.py`.

| Algorithm | Crate | Status document | Result |
|---|---|---|---|
| SHA-3 | `crates/algorithms/sha3/` | [status](crates/algorithms/sha3/proofs/verification_status.md) | functional correctness [1] |
| ML-KEM | `libcrux-ml-kem/` | [status](libcrux-ml-kem/proofs/ml_kem_verification_status.md) | functional correctness [1] |
| ML-DSA | `libcrux-ml-dsa/` | [status](libcrux-ml-dsa/proofs/verification_status.md) | mixed [2] |
| KMAC | `crates/algorithms/kmac/` | none | runtime safety only |

[1] Functional correctness against the Hacspec specification (SHA-3:
`specs/sha3`; ML-KEM: `specs/ml-kem`). The status document gives the per-function
result.

[2] The core arithmetic and serialization code is proven functionally correct
against the Hacspec specification (`specs/ml-dsa`); the remaining higher-level
code is verified for runtime safety. The status document gives the per-function
result.
